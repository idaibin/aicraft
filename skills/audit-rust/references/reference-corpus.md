## Explicit Local Governance Decisions

- Apply the rule priority in `SKILL.md`.
- Require targeted repository and analogous-code investigation before creating
  public traits, global state, runtimes, pools, caches, or database layers.
- Distinguish portable governance from organization baselines, new-project
  templates, repository contracts, and legacy exceptions.
- Treat version numbers and directory trees as scoped baselines or templates,
  not universal Rust rules.
- Prefer local-first, single-binary, minimal-dependency designs when they already
  satisfy the product contract.
- Treat file length, `clone`, `Arc<Mutex<_>>`, `unwrap`, unsafe, and full scans as
  evidence prompts, not automatic findings.
- Require architecture, migration, performance, and operational documentation to
  change with the source-of-truth boundary.
- Require add, reuse, move, rename, and delete operations to update the complete
  affected manifest, registration, test, migration, deployment, docs, and index
  chain, while preserving historical migrations needed by installed versions.

These are reusable engineering governance decisions supported by the corpus,
not claims that Rust or SQLite mandates one folder structure or framework.
